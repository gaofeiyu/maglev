# 任务: BMAD 帮助

## 关键规则

- **空的 `phase` = 随时** — 通用工具无论工作流状态如何均可工作
- **编号的阶段表示顺序** — 像 `1-discover` → `2-define` → `3-build` → `4-ship` 这样的阶段按顺序流动（命名因模块而异）
- **留在模块中** — 根据阶段+顺序排序，引导完成活动模块的工作流
- **描述包含路由** — 阅读备用路径（例如，“如果需要修复，请返回上一步”）
- **`required=true` 阻止进度** — 必须完成所需的工作流才能进入后续阶段
- **工件揭示完成情况** — 在解析的输出路径中搜索 `outputs` 模式，将找到的文件模糊匹配到工作流记录

## 模块检测

- **空的 `module` 列** → 通用工具（适用于所有模块）
- **命名的 `module`** → 模块特定工作流

从对话上下文、最近的工作流或用户查询关键字中检测活动模块。如果含糊不清，请询问用户。

## 输入分析

确定刚刚完成了什么：
- 是否有人声明他们完成了某事？就像那是输入一样继续。
- 此对话中刚刚完成了一个工作流吗？就像那是输入一样继续。
- 在解析的工件位置搜索文件；模糊匹配工作流 `outputs` 模式。
- 如果存在 `index.md`，请阅读它以获取更多上下文。
- 如果仍然不清楚，请问：“你最近完成了哪个工作流？”

## 执行

1. **加载目录** — 加载 `{project-root}/_bmad/_config/bmad-help.csv`

2. **解析输出位置** — 扫描 `_bmad/` 下的每个文件夹（`_config` 除外）以查找 `config.yaml`。对于每个工作流记录，根据该模块的配置解析其 `output-location` 变量，以便可以搜索工件路径。

3. **分析输入** — 任务可能提供工作流名称/代码、会话短语或什么都不提供。使用上面的 INPUT ANALYSIS 推断刚刚完成了什么。

4. **检测活动模块** — 使用上面的 MODULE DETECTION 确定用户正在哪个模块中工作。

5. **提出建议** — 根据完成的工作流、阶段/顺序排序（KEY RULES）和工件检测显示后续步骤。按照以下格式：

## 推荐的输出格式

   **可选项目优先** — 列出可选工作流，直到达到所需步骤
   **必需项目其次** — 列出下一个必需的工作流
   对于每个项目显示：
   - 工作流 **名称**
   - **命令** (以 `/` 为前缀，例如 `/bmad:example:build-prototype`)
   - 來自 CSV 的 **Agent** 标题和显示名称（例如，“🎨 Alex (设计师)”）
   - 简要 **描述**

   ### 额外的响应输出指导以传达：
   - 在 **新的上下文窗口** 中运行每个工作流
   - 使用 (`/` + `agent-command`) 加载 agent，或直接运行工作流命令
   - 对于 **验证工作流**: 如果可用，建议使用不同的高质量 LLM
   - 对于会话请求：在清晰呈现的同时匹配用户的语气

6. 在提出建议后返回到调用进程。
