## 上下文边界：

- 来自先前步骤的当前文档和 frontmatter 可用
- 来自 step-02 的项目类型可用于创新信号匹配
- 项目类型 CSV 数据将在此步骤中加载
- 专注于检测真正的创新，而不是强迫创造力

## 可选步骤检查：

在继续此步骤之前，扫描创新信号：

- 倾听诸如“没有什么像这样”、“重新思考 X 如何工作”之类的语言
- 从 CSV 检查项目类型创新信号
- 寻找新颖的方法或独特的组合
- 如果未检测到创新，跳过此步骤

## 你的任务：

检测并探索产品中的创新模式，专注于使其真正新颖的因素以及如何验证创新方面。

## 创新发现序列：

### 1. 加载项目类型创新数据

加载特定于此项目类型的创新信号：

- 完整加载 `{projectTypesCSV}`
- 查找 `project_type` 匹配来自 step-02 的检测类型的行
- 提取 `innovation_signals`（分号分隔列表）
- 提取 `web_search_triggers` 用于潜在的创新研究

### 2. 倾听创新指标

监控对话中的一般和特定于项目类型的创新信号：

#### 一般创新语言：

- "没有什么像这样存在"
- "我们正在重新思考 [X] 如何运作"
- "首次将 [A] 与 [B] 结合"
- "解决 [问题] 的新颖方法"
- "以前没有人做过 [概念]"

#### 项目类型特定信号 (来自 CSV):

将用户描述与针对其 project_type 的 innovation_signals 进行匹配：

- **api_backend**: "API composition;New protocol"
- **mobile_app**: "Gesture innovation;AR/VR features"
- **saas_b2b**: "Workflow automation;AI agents"
- **developer_tool**: "New paradigm;DSL creation"

### 3. 初步创新筛选

提出有针对性的创新发现问题：
- 引导探索是什么让产品具有创新性
- 探索他们是否在挑战现有的假设
- 询问技术/方法的某些新颖组合
- 识别以前未做过的事情
- 了解哪些方面感觉最具创新性

### 4. 深度创新探索 (如果检测到)

如果发现创新信号，深入探索：

#### 创新发现问题：
- 与现有解决方案相比，它有什么独特之处？
- 你正在挑战什么假设？
- 我们如何验证它有效？
- 如果不行，后备方案是什么？
- 以前有人尝试过这个吗？

#### 市场背景研究：

如果检测到相关创新，考虑网络搜索背景：
使用来自项目类型 CSV 的 `web_search_triggers`：
`[web_search_triggers] {concept} innovations {date}`

### 5. 生成创新内容 (如果检测到创新)

准备要追加到文档的内容：

#### 内容结构：

保存到文档时，追加这些 2 级和 3 级部分：

```markdown
## 创新与新颖模式

### 检测到的创新领域

[基于对话识别出的创新模式]

### 市场背景与竞争格局

[基于对话的市场背景与研究]

### 验证方法

[基于对话的验证方法论]

### 风险缓解

[基于对话的创新风险与后备方案]
```

### 6. 展示菜单选项 (仅当检测到创新时)

展示创新内容以供审查，然后显示菜单：
- 显示已识别的创新方面（使用第 5 部分的结构）
- 强调与现有解决方案的差异
- 询问他们是否愿意进一步完善、获得其他观点或继续
- 作为对话的一部分自然地展示菜单选项

显示： "**选择：** [A] 高级诱导 [P] 派对模式 [C] 继续到项目类型分析 (第 7 步，共 11 步)"

显示： "**选择：** [A] 高级诱导 - 让我们尝试寻找创新角度 [C] 继续 - 跳过创新部分并移动到项目类型分析 (第 7 步，共 11 步)"
