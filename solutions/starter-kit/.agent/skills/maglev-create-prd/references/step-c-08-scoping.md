## 范围界定序列：

### 1. 审查当前 PRD 状态

分析目前记录的所有内容：
- 展示既定愿景、成功标准、旅程的综合
- 评估领域和创新重点
- 评估范围影响：简单的 MVP、中等或复杂项目
- 询问初步评估感觉是否正确，或者他们是否有不同的看法

### 2. 定义 MVP 策略

促进战略 MVP 决策：
- 探索 MVP 哲学选项：问题解决、体验、平台或收入 MVP
- 提出关键问题：
  - 什么是能让用户说“这很有用”的最低限度？
  - 什么会让投资者/合作伙伴说“这有潜力”？
  - 验证学习的最快途径是什么？
- 引导他们选择适合其产品的 MVP 方法

### 3. 范围决策框架

使用结构化决策来确定范围：

**必须有 (Must-Have) 分析：**
- 引导识别绝对的 MVP 必需品
- 对于每个旅程和成功标准，询问：
  - 没有这个，产品会失败吗？
  - 这最初可以是手动的吗？
  - 这对早期采用者来说是一个破坏性因素吗？
- 分析旅程中的 MVP 必需品

**锦上添花 (Nice-to-Have) 分析：**
- 识别稍后可以添加的内容：
  - 增强但不是必不可少的功能
  - 可以稍后添加的用户类型
  - 基于 MVP 构建的高级功能
- 询问可以在版本 2、3 等中添加哪些功能

### 4. 渐进式功能路线图

创建分阶段开发方法：
- 引导跨开发阶段的功能映射
- 结构化为第 1 阶段 (MVP)、第 2 阶段 (增长)、第 3 阶段 (愿景)
- 确保清晰的进展和依赖关系

- 核心用户价值交付
- 基本用户旅程
- 可靠运行的基本功能

**第 2 阶段：增长**

- 额外的用户类型
- 增强功能
- 规模改进

**第 3 阶段：扩展**

- 高级能力
- 平台功能
- 新市场或用例

**你目前的愿景在这一开发顺序中处于什么位置？**"

### 5. 基于风险的范围界定

识别并缓解范围风险：

**技术风险：**
"看着你的创新和领域要求：

- 技术上最具挑战性的方面是什么？
- 我们可以简化初始实现吗？
- 关于技术可行性最危险的假设是什么？"

**市场风险：**

- 最大的市场风险是什么？
- MVP 如何解决这个问题？
- 我们需要什么学习来消除这个风险？"

**资源风险：**

- 如果我们的资源比计划的少怎么办？
- 需要的绝对最小团队规模是多少？
- 我们可以用更小的功能集发布吗？"
