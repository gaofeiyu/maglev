# 人员转型与技能重塑指南 (Workforce Transition & Upskilling Guide)

本文档旨在消除焦虑，为团队中现有的普适性人才提供一条清晰的**职业升级路径**。
AI 不是来取代你的，而是来强迫你升级的。请根据你现在的角色，找到你的转型地图。

---

## 1. 传统产品经理 (PM) -> Value Owner (VO)
*告别“文档写手”，成为“意图架构师”*

### 1.1 Stop & Start 清单
| 🚫 停止做 (Stop Doing) | ✅ 开始做 (Start Doing) |
| :--- | :--- |
| **写长篇 PRD**: 没人看，AI 也读不懂冗长的废话。 | **写结构化 Markdown**: 学习 YAML frontmatter，用树状清单描述逻辑。 |
| **手画高保真原型**: 浪费时间，且容易误导视觉设计。 | **截图 + 文字 Prompt**: 找竞品截图，用文字告诉 AI “像这个，但是...” |
| **当传声筒**: 客户说什么就记什么。 | **当过滤器**: 学会说“不”，只把 20% 真正有价值的需求转化为 Spec。 |
| **用 Word/Excel**: 信息的孤岛。 | **用 Git/IDE**: 即使只改一个标点，也要在 Repo 里提 Commit。 |

### 1.2 转型第一周·行动计划
1.  **Day 1**: 卸载/隐藏 Word，安装 **VS Code / Cursor**。
2.  **Day 2**: 学习 Markdown 语法 (标题、列表、代码块、链接)。
3.  **Day 3**: 学习 Git 基础 (`clone`, `pull`, `commit`, `push`)。
4.  **Day 4**: 尝试把你手头的一个需求，拆解成 `specs/` 目录下的一个 `.md` 文件。
5.  **Day 5**: 尝试用 AI 润色你的需求文档，看 AI 能否读懂。

## 2. 传统研发 (Dev) -> Tech Pilot (TP)
*告别“CRUD 搬砖”，成为“AI 驯化师”*

### 2.1 Stop & Start 清单
| 🚫 停止做 (Stop Doing) | ✅ 开始做 (Start Doing) |
| :--- | :--- |
| **手写样板代码**: HTML 结构、CSS 类名、简单的增删改查。 | **Prompt 生成**: 用自然语言描述意图，让 AI 生成骨架，然后微调。 |
| **百度/谷歌搜索**: 遇到报错先搜，然后在一个个点开看。 | **把报错喂给 AI**: 直接粘错误堆栈给 AI，让它分析原因并给修复方案。 |
| **沉迷技术细节**: 纠结于虽然不重要但很难实现的算法。 | **拥抱 MVP 思维**: “能不能先用个现成的库顶一下？” |
| **只管后端/前端**: “这个 CSS 我不会调，找前端”。 | **全栈尝试**: 在 AI 加持下，没有你不懂的语言，大胆跨界。 |

### 2.2 转型第一周·行动计划
1.  **Day 1**: 安装并配置好 AI 编程助手 (Cursor / Copilot)。
2.  **Day 2**: 强迫自己**不写一行代码**完成一个简单功能（全靠 Prompt）。
3.  **Day 3**: 学习 **Prompt Engineering** (Context 引用, Chain of Thought)。
4.  **Day 4**: 做一次 Code Review，专门挑 AI 生成代码的毛病（变量命名、安全隐患）。
5.  **Day 5**: 尝试解决一个你从未接触过的技术栈的问题 (e.g., 后端去改 CSS)。

## 3. 传统测试/设计 (QA/UI) -> Experience Guardian (XG)
*告别“机械执行”，成为“品质把关人”*

### 3.1 Stop & Start 清单
| 🚫 停止做 (Stop Doing) | ✅ 开始做 (Start Doing) |
| :--- | :--- |
| **只画静态图**: 交付给研发后就不管了，上线再验收。 | **参与代码构建**: 在 Preview 环境中实时调整 CSS 变量 (Design Tokens)。 |
| **机械执行用例**: 按照 Excel 表格一个个点。 | **探索性破坏**: 像黑客一样思考，尝试搞崩系统，尝试生僻路径。 |
| **只提 Bug**: 截张图发在群里说“有问题”。 | **提可复现 Issue**: 在 `issues/` 目录下用 Markdown 记录复现步骤和 Spec 引用。 |
| **等待排期**: 这种小体验问题没人修就算了。 | **自己动手**: 对于简单的文案/颜色/间距问题，直接修代码提 PR。 |

### 3.2 转型第一周·行动计划(
1.  **Day 1**: 获取代码仓库权限，把项目跑起来 (LocalHost)。
2.  **Day 2**: 熟悉 `design/tokens` 文件，知道颜色、字体都在哪里定义。
3.  **Day 3**: 学习如何用浏览器 DevTools 实时修改样式。
4.  **Day 4**: 尝试提一个标准的 Markdown Issue，包含截图和复现步骤。
5.  **Day 5**: 尝试修复一个简单的 UI Bug，并提交 Merge Request。

## 4. 心理建设 (Must-Have Mindset)

- **别怕 AI**: AI 写的代码比你快，但不一定比你好。你的价值在于**判断**。
- **别怕犯错**: 在转型期，允许因为不熟悉工具而变慢。现在的慢是为了将来的极速。
- **拥抱透明**: 全资产同仓意味着你的每一步操作都被看见。这不是监控，这是**协作的基础**。
