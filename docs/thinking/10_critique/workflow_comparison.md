# 新旧范式对比：工作流 (Workflow Comparison)

本文件对比了传统瀑布/敏捷开发模式与 AI 驱动的协作模式的主要差异。

## 1. 核心流程对比

| 阶段 | 传统模式 (Traditional) | AI 驱动新范式 (AI-Driven) |
| :--- | :--- | :--- |
| **需求阶段** | PM 撰写文档 -> 评审 -> 修改 | PM + AI 快速生成原型/MVP -> 体验 -> 调整 |
| **设计阶段** | UI/UX 详细设计 -> 交付切图 | AI 生成设计系统/组件 -> 实时预览调整 |
| **开发阶段** | 手写代码 -> 查文档 -> 调试 | Prompt -> AI 生成代码 -> 人工 Review -> 修正 |
| **测试阶段** | 编写用例 -> 执行 -> 提 Bug -> 修复 | AI 生成用例 & 自动修复简单 Bug -> 人工探索性测试 |
| **交付阶段** | 固定周期发布 | 持续交付，颗粒度更细 |

## 2. 协作模式差异

### 传统模式：线性接力
- **特征**：环节分明，文档驱动，信息在传递中衰减。
- **痛点**：沟通成本高，等待时间长，理解偏差大。
- **口号**："按照文档做。"

### AI 新范式：环形迭代
- **特征**：角色边界模糊，原型驱动，信息在上下文中增强。
- **优势**：反馈周期极短（分钟级），验证成本低。
- **口号**："看，这是不是你想要的？"

## 3. 信息流转
- **传统**: 需求文档 (Word/Wiki) -> 代码 (Git) -> 测试报告 (Excel/Jira)
- **新范式**: Context (已验证的知识/代码片/Prompt) 贯穿始终。文档不仅是给人看的，更是给 AI 读的（作为 Context）。
